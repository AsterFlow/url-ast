# Guia de Migração

Esta página cobre a atualização entre versões maiores do url-ast. Comece pela seção da versão de onde você está **saindo**.

## Migrando para a v4.1 (navegador / edge)

A v4.1 adiciona uma **entrada de navegador** para que a engine Rust/WASM rode em navegadores, bundlers e runtimes edge — sem nenhum componente Node envolvido.

<Callout type="info">
  **Consumidores Node não são afetados.** A entrada Node continua inicializando o WASM de forma síncrona no import, então o código v4.0 existente continua funcionando sem mudanças. Esta seção só importa se você tem como target um navegador, um bundler ou um runtime edge.
</Callout>

### Importe de `url-ast/browser`

Ao ter como target navegadores, bundlers ou runtimes edge, importe da entrada `url-ast/browser` em vez da raiz do pacote:

```typescript
import { Analyze, initWasm } from 'url-ast/browser'
```

### Inicialize o WASM de forma assíncrona

Diferente da entrada Node, a entrada de navegador **não** inicializa o WASM automaticamente. Chame `await initWasm()` uma vez, antes do seu primeiro parse:

```typescript
import { Analyze, initWasm } from 'url-ast/browser'

// Uma vez, na inicialização — antes de qualquer parsing
await initWasm()

const template = new Analyze('/users/:id.number')
```

`initWasm(input?)` aceita uma fonte WASM opcional (uma URL, `Response`, bytes ou um módulo compilado). **Sem argumento**, ele resolve `new URL('../wasm/wasm_bg.wasm', import.meta.url)` como um asset de bundler — a maioria dos bundlers reconhece isso automaticamente.

<Callout type="info">
  `initWasm` e `isWasmAvailable` também são reexportados na raiz do pacote, então você pode acessá-los a partir de `url-ast` sem trocar de entrada.
</Callout>

### `getBuffer()` retorna `Uint8Array`

A serialização deixou de usar a API `Buffer` (exclusiva do Node) e passou para tipos padrão da web. `getBuffer()` agora retorna um `Uint8Array` (antes descrito como um `Buffer` do Node). Um `Buffer` *é* um `Uint8Array`, então a maioria dos consumidores não é afetada — mas atualize qualquer código de serialização que assumia especificamente um `Buffer` do Node (por exemplo, chamando métodos exclusivos de `Buffer` no resultado).

### Checklist de migração

- [ ] Para targets de navegador / bundler / edge, importe de `url-ast/browser`
- [ ] Chame `await initWasm()` uma vez antes do parsing (apenas na entrada de navegador)
- [ ] Atualize o código de serialização que dependia de `getBuffer()` retornar um `Buffer` do Node
- [ ] Código Node não precisa de mudanças

## Migrando da v3 para a v4

A v4 troca a engine de parsing por um **núcleo Rust compilado para WebAssembly**. O scanner e o construtor da AST agora rodam em WASM; a camada TypeScript é uma ponte fina que decodifica o buffer retornado pela engine.

<Callout type="info">
  **Nenhuma mudança de código é necessária.** A sintaxe de templates e a API pública (`Analyze`, `AST`, `parse`, `parseAndAnalyze`, `ErrorLog`) são idênticas à v3. É uma troca de engine — a maioria dos projetos atualiza apenas subindo a versão.
</Callout>

### O que realmente mudou

| Aspecto | v3 | v4 |
| :--- | :--- | :--- |
| Engine de parsing | TypeScript (esbuild, ES2024) | Rust → WebAssembly |
| Artefato de runtime | Apenas JS | Ponte JS + `dist/wasm/wasm_bg.wasm` |
| Init do WASM | — | Síncrono no import via `node:fs` |
| Sintaxe de template / API pública | — | **Inalterada** |

### Requisitos de runtime

O módulo WASM é carregado de forma síncrona quando você faz `import 'url-ast'`, lendo `dist/wasm/wasm_bg.wasm` do disco com `node:fs`.

- **Node.js ≥ 20** é obrigatório (inalterado em relação à v3).
- O build tem **target Node**. Não é feito para rodar diretamente no navegador como publicado.

### Notas sobre bundlers

Como o pacote lê um asset `.wasm` do disco no momento do import, os bundlers precisam manter esse asset acessível:

- **Bundlers server / Node (esbuild, tsup, Next.js server):** trate `node:fs` e `node:url` como externos/nativos e garanta que `dist/wasm/wasm_bg.wasm` seja copiado ao lado do bundle emitido. A maioria dos targets Node faz isso por padrão.
- **Serverless / edge:** inclua `dist/wasm/wasm_bg.wasm` no bundle de deploy (ex.: `outputFileTracingIncludes` do Next.js, ou a configuração de inclusão de assets da sua plataforma).
- Se o import não localizar o binário, o url-ast expõe `isWasmAvailable` (booleano) para você detectar cedo uma engine não resolvida.

### Checklist de migração

- [ ] Suba o `url-ast` para `^4.0.0`
- [ ] Confirme que o runtime é Node.js ≥ 20
- [ ] Garanta que seu bundler entrega `dist/wasm/wasm_bg.wasm` junto ao bundle
- [ ] Nenhuma mudança de template ou API é necessária — o código v3 existente compila como está

## Migrando da v2 para a v3

Esta seção cobre as **breaking changes** introduzidas no url-ast v3 e como adaptar seu código existente.

<Callout type="warning">
  A v3 redesenha a AST e altera a sintaxe de anotação de tipos. A migração exige mudanças no código que usa templates.
</Callout>

### Mudança principal: `.type` em vez de `=type`

A maior breaking change da v3 é a migração da anotação de tipos do operador `=` para o operador `.`:

```diff
- const template = new Analyze('/users/:id=number')
+ const template = new Analyze('/users/:id.number')
```

O operador `=` agora é usado **exclusivamente** para definir valores default:

```typescript
// v3: `.` para tipo, `=` para default
const template = new Analyze('/search?page.number=1&limit.number=10')
```

### Novas funcionalidades na v3

### Operador `~` (opcional)

Parâmetros agora podem ser marcados como opcionais:

```typescript
const template = new Analyze('/users/:id/profile/:~tab')
```

### Suporte a `enum`

Novo tipo para restringir valores:

```typescript
const template = new Analyze('/theme/:mode.enum[dark,light]')
```

### Enums genéricos

```typescript
const template = new Analyze('/status/:state.enum')
```

### AST redesenhada

A v3 redesenhou completamente a estrutura da AST:

| Aspecto | v2 | v3 |
| :--- | :--- | :--- |
| Abstração | Granular (espelhava tokens) | Semântica (agrupada por significado) |
| Delimitadores | `Delimiters`, `EncodingSymbols` | `ParameterDelimiters`, `DynamicVariableDelimiters`, `GeneralDelimiters`, `Operators` |
| Expressões | `Null`, `Value` | `None`, `Type`, `Default`, novos estados para dinâmicos e catch-all |

### Renomeações de expressões internas

Os valores do enum `InternalExpression` foram renomeados:

```diff
- InternalExpression.Null
+ InternalExpression.None

- InternalExpression.Value
+ InternalExpression.Type
```

### Build

A v3 migrou o build para **esbuild**, com targets CJS e ESM e target **ES2024**.

### Checklist de migração

- [ ] Substituir `=type` por `.type` em todos os templates
- [ ] Usar `=` somente para valores default (opcionalmente com `.type`)
- [ ] Renomear `displayErrors()` para `formatErrors()` em `Analyze`
- [ ] Renomear `display()` para `toString()` em `AST` e `ErrorLog`
- [ ] Atualizar imports de tipos se usava enums internos diretamente
- [ ] Verificar se seu target de build suporta ES2024

<Callout type="info">
  Consulte o [CHANGELOG completo](https://github.com/AsterFlow/url-ast/blob/main/CHANGELOG.md) para a lista detalhada de todas as mudanças.
</Callout>
