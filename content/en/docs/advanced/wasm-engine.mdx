# Rust/WASM Engine

Since **v4**, url-ast's parsing core is written in **Rust** and compiled to **WebAssembly**. This page explains how that pipeline works and what it means for consuming the library.

<Callout type="info">
  This is an internals page. You do not need any of it to use url-ast ‚Äî the engine
  initializes itself on `import`. Read on if you bundle for unusual targets or want
  to understand the architecture.
</Callout>

## Why WASM?

URL parsing is a tight, allocation-heavy loop. Moving the scanner and AST builder into a Rust core removes per-token JavaScript overhead and lets the hot path run as compact WebAssembly bytecode. The TypeScript layer stays ‚Äî it just becomes a **thin bridge** that hands input to the engine and decodes what comes back.

## How the pieces fit

```
Rust crate (wasm/, wasm/bridge.rs)
      ‚îÇ  cargo build --target wasm32-unknown-unknown
      ‚ñº
wasm-bindgen --target web        ‚Üí  glue (wasm.js) + wasm_bg.wasm
      ‚îÇ  wasm-opt -Oz
      ‚ñº
dist/wasm/  (wasm.js, wasm.d.ts, wasm_bg.wasm)
      ‚îÇ  inlined glue, raw binary loaded at runtime
      ‚ñº
TypeScript bridge (src/wasmBridge.ts)  ‚Üí  Node[] + ErrorLog[]
```

- **Build:** `scripts/build-wasm.sh` runs `cargo build` for `wasm32-unknown-unknown`, then `wasm-bindgen --target web`, then `wasm-opt -Oz` for size. `wasm32-unknown-unknown` is architecture-independent, so a single artifact serves every platform ‚Äî no arm64/x86_64 split.
- **Distribution:** the output lands in `dist/wasm/`. The bindgen glue is inlined into the JS bundle at build time; only the raw `wasm_bg.wasm` binary is loaded at runtime.

## Initialization

The engine is initialized **synchronously at import time**. When you `import 'url-ast'`, the entry module resolves `wasm_bg.wasm` relative to itself and loads it with `node:fs`:

```typescript
// simplified from src/initWasmSync.ts
import { readFileSync } from 'node:fs'
import { initSync } from 'wasm/wasm'

initSync({ module: readFileSync(pathToWasmBinary) })
export let isWasmAvailable = /* true once initialized */
```

Because it is synchronous, `Analyze`, `AST`, and `parse` are usable immediately after import ‚Äî there is **no async `init()` to await**.

<Callout type="info">
  The default entry is **Node-targeted** and loads the binary via `node:fs` at import
  (Node.js **‚â• 20**). Since **v4.1** a browser/edge entry is also published ‚Äî see
  [Browser & edge](#browser--edge) below.
</Callout>

### Detecting availability

If the binary cannot be located (an aggressive bundler dropped the asset, for example), initialization fails quietly and the exported `isWasmAvailable` flag stays `false`:

```typescript
import { isWasmAvailable } from 'url-ast'

if (!isWasmAvailable) {
  throw new Error('url-ast WASM engine failed to initialize ‚Äî check that dist/wasm/wasm_bg.wasm ships with your bundle')
}
```

## The bridge buffer

A parse call returns a view over the engine's linear memory ‚Äî **no copy out**. The bridge decodes that shared buffer synchronously before any further WASM call can invalidate it. The layout is:

- `u16` root count, followed by the root nodes (fixed-size records, read recursively)
- `u16` error count, followed by each error as `u16 codeLen, code, u16 msgLen, msg, u32 start, u32 end`

The decoded result is a plain `{ nodes: Node[]; errors: ErrorLog[] }` ‚Äî the same shape the rest of the API consumes, so nothing downstream is aware the work happened in Rust.

## Browser & edge

Since **v4.1**, a browser entry runs the same engine with **no Node components**. It
does not read from `node:fs` and does not auto-initialise ‚Äî you call an async
`initWasm()` once before using the API:

```ts
import { initWasm, Analyze } from 'url-ast/browser'
import wasmUrl from 'url-ast/wasm/wasm_bg.wasm?url'

await initWasm(wasmUrl)                 // load + instantiate the WASM binary
new Analyze('/users/:id.number')
```

`initWasm(input?)` accepts a URL/string to `fetch`, a `Response`, raw bytes, or a
pre-compiled `WebAssembly.Module`. With no argument it defaults to
`new URL('wasm_bg.wasm', import.meta.url)`. It is safe to call repeatedly ‚Äî later
calls resolve immediately once the engine is ready.

The serialization layer was also moved off the Node-only `Buffer` API onto
`Uint8Array` / `DataView`, so `getBuffer()` / `fromBuffer()` work in the browser too.
`package.json` conditional exports route Node to the synchronous build and bundlers
to this one automatically, so `import 'url-ast'` keeps working on both.

<Callout type="info">

</Callout>
