# Migration Guide

This page covers upgrading between major versions of url-ast. Start with the section for the version you are moving **from**.

## Migrating to v4.1 (browser / edge)

v4.1 adds a **browser entry** so the Rust/WASM engine can run in browsers, bundlers and edge runtimes ŌĆö no Node components involved.

<Callout type="info">
  **Node.js consumers are unaffected.** The Node entry still initializes WASM synchronously on import, so existing v4.0 code keeps working with no changes. This section only matters if you target a browser, a bundler, or an edge runtime.
</Callout>

### Import from `url-ast/browser`

When targeting browsers, bundlers or edge runtimes, import from the `url-ast/browser` entry instead of the package root:

```typescript
import { Analyze, initWasm } from 'url-ast/browser'
```

### Initialize WASM asynchronously

Unlike the Node entry, the browser entry does **not** auto-initialize WASM. Call `await initWasm()` once, before your first parse:

```typescript
import { Analyze, initWasm } from 'url-ast/browser'

// Once, at startup ŌĆö before any parsing
await initWasm()

const template = new Analyze('/users/:id.number')
```

`initWasm(input?)` accepts an optional WASM source (a URL, `Response`, bytes or a compiled module). With **no argument**, it resolves `new URL('../wasm/wasm_bg.wasm', import.meta.url)` as a bundler asset ŌĆö most bundlers pick this up automatically.

<Callout type="info">
  `initWasm` and `isWasmAvailable` are re-exported from the package root as well, so you can reach them from `url-ast` without switching entries.
</Callout>

### `getBuffer()` returns `Uint8Array`

Serialization moved off the Node-only `Buffer` API onto web-standard types. `getBuffer()` now returns a `Uint8Array` (previously described as a Node `Buffer`). A `Buffer` *is* a `Uint8Array`, so most callers are unaffected ŌĆö but update any serialization code that assumed a Node `Buffer` specifically (for example, calling `Buffer`-only methods on the result).

### Migration checklist

- [ ] For browser / bundler / edge targets, import from `url-ast/browser`
- [ ] Call `await initWasm()` once before parsing (browser entry only)
- [ ] Update serialization code that relied on `getBuffer()` returning a Node `Buffer`
- [ ] Node.js code needs no changes

## Migrating from v3 to v4

v4 swaps the parsing engine for a **Rust core compiled to WebAssembly**. The scanner and AST builder now run in WASM; the TypeScript layer is a thin bridge that decodes the buffer the engine returns.

<Callout type="info">
  **No source changes are required.** The template syntax and the public API (`Analyze`, `AST`, `parse`, `parseAndAnalyze`, `ErrorLog`) are identical to v3. This is an engine swap ŌĆö most projects upgrade by bumping the version.
</Callout>

### What actually changed

| Aspect | v3 | v4 |
| :--- | :--- | :--- |
| Parsing engine | TypeScript (esbuild, ES2024) | Rust ŌåÆ WebAssembly |
| Runtime artifact | JS only | JS bridge + `dist/wasm/wasm_bg.wasm` |
| WASM init | ŌĆö | Synchronous at import via `node:fs` |
| Template syntax / public API | ŌĆö | **Unchanged** |

### Runtime requirements

The WASM module is loaded synchronously when you `import 'url-ast'`, reading `dist/wasm/wasm_bg.wasm` from disk with `node:fs`.

- **Node.js Ōēź 20** is required (unchanged from v3).
- The build is **Node-targeted**. It is not meant to run directly in a browser as shipped.

### Bundler notes

Because the package reads a `.wasm` asset from disk at import time, bundlers must keep that asset reachable:

- **Server / Node bundlers (esbuild, tsup, Next.js server):** treat `node:fs` and `node:url` as external/native, and make sure `dist/wasm/wasm_bg.wasm` is copied next to the emitted bundle. Most Node targets do this by default.
- **Serverless / edge:** include `dist/wasm/wasm_bg.wasm` in the deployment bundle (e.g. Next.js `outputFileTracingIncludes`, or your platform's asset-inclusion setting).
- If import fails to locate the binary, url-ast exposes `isWasmAvailable` (a boolean) so you can detect an unresolved engine early.

### Migration checklist

- [ ] Bump `url-ast` to `^4.0.0`
- [ ] Confirm the runtime is Node.js Ōēź 20
- [ ] Ensure your bundler ships `dist/wasm/wasm_bg.wasm` alongside the bundle
- [ ] No template or API changes needed ŌĆö existing v3 code compiles as-is

## Migrating from v2 to v3

This section covers the **breaking changes** introduced in url-ast v3 and how to adapt your existing code.

<Callout type="warning">
  v3 redesigns the AST and changes the type annotation syntax. Migration requires changes to code that uses templates.
</Callout>

### Main change: `.type` instead of `=type`

The biggest breaking change in v3 is the migration of type annotation from the `=` operator to the `.` operator:

```diff
- const template = new Analyze('/users/:id=number')
+ const template = new Analyze('/users/:id.number')
```

The `=` operator is now used **exclusively** for defining default values:

```typescript
// v3: `.` for type, `=` for default
const template = new Analyze('/search?page.number=1&limit.number=10')
```

### New features in v3

### `~` operator (optional)

Parameters can now be marked as optional:

```typescript
const template = new Analyze('/users/:id/profile/:~tab')
```

### `enum` support

New type to restrict values:

```typescript
const template = new Analyze('/theme/:mode.enum[dark,light]')
```

### Generic enums

```typescript
const template = new Analyze('/status/:state.enum')
```

### AST redesign

v3 completely redesigned the AST structure:

| Aspect | v2 | v3 |
| :--- | :--- | :--- |
| Abstraction | Granular (mirrored tokens) | Semantic (grouped by meaning) |
| Delimiters | `Delimiters`, `EncodingSymbols` | `ParameterDelimiters`, `DynamicVariableDelimiters`, `GeneralDelimiters`, `Operators` |
| Expressions | `Null`, `Value` | `None`, `Type`, `Default`, new states for dynamic and catch-all |

### Internal expression renames

The `InternalExpression` enum values were renamed:

```diff
- InternalExpression.Null
+ InternalExpression.None

- InternalExpression.Value
+ InternalExpression.Type
```

### Build

v3 migrated the build to **esbuild**, with CJS and ESM targets and **ES2024** target.

### Migration checklist

- [ ] Replace `=type` with `.type` in all templates
- [ ] Use `=` only for default values (optionally with `.type`)
- [ ] Rename `displayErrors()` to `formatErrors()` on `Analyze`
- [ ] Rename `display()` to `toString()` on `AST` and `ErrorLog`
- [ ] Update type imports if you were using internal enums directly
- [ ] Verify that your build target supports ES2024

  See the [full CHANGELOG](https://github.com/AsterFlow/url-ast/blob/main/CHANGELOG.md) for a detailed list of all changes.
</Callout>
